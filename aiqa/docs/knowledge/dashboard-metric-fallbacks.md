# Fallback и риски метрик при неполных ALM-данных

Как вести себя **визуально и на обзорах**, если кастомные поля не созданы, пустые, или Analytics недоступен. Детали полей: [alm-required-fields.md](alm-required-fields.md). Общая схема дашборда: [ado-dashboard-setup.md](ado-dashboard-setup.md).

---

## 1. `Custom.FoundStage` отсутствует или пусто у большинства багов

**Симптом:** chart «Bugs by found_stage» пустой, один столбец `Empty`, или запрос падает при сохранении.

**Действия без блокировки демо:**

1. Оставить виджеты **Pre-prod Bugs (MTD)** и **Prod Bugs (MTD)** как есть — они явно фильтруют по стадии; при пустом поле счётчики будут низкими или нулевыми.
2. На chart **All Confirmed Bugs With Stage** временно добавить в заголовок пометку: `(data gap: FoundStage)` и вынести в список рисков задачи.
3. Для обзора использовать **список**: Query Results по запросу с `Custom.FoundStage` в SELECT — видно пустые значения построчно.
4. Strict **Bugs per PBI** по pipeline не заменяет found_stage; это отдельный сигнал качества связей и BugType.

**Не делать:** выключать chart без замены — лучше явный «пустой сигнал», чем исчезнувший виджет.

---

## 2. `Custom.BugType` не заполнен или NOT_SET

**Симптом:** Legacy / New счётчики занижены, метрики «легаси» нечестны.

**Действия:**

1. В описании дашборда зафиксировать: **распределение New vs Legacy недостоверно** до нормализации поля.
2. Для операционного контроля временно опираться на **state/priority** charts и prod/pre-prod счётчики по found_stage (если оно заполнено лучше BugType).
3. Запустить `check_bug_fields.py` после появления PAT — приложить краткий вывод в комментарий к задаче.

---

## 3. `Custom.QaDecision` пустой у закрытых Feature

**Симптом:** счётчики sign-off высокие, список нарушителей длинный — это **корректный сигнал**, не баг дашборда.

**Действия:**

1. Не скрывать виджеты — они показывают дисциплину.
2. На обзоре договориться о **deadline заполнения** ретроспективно для уже закрытых (или исключить период из KPI в комментарии).

---

## 4. Нет явной связи Bug → Feature (`System.Parent` пустой)

**Симптом:** strict Bugs per PBI даёт `NO_DATA` или низкий числитель при «много багов визуально».

**Действия:**

1. Считать это **блокером интерпретации** «багов на поставку», а не сбоем пайплайна.
2. В Run Summary / artifact явно читать строку **DataStatus** и переносить в комментарий к релизу.

---

## 5. Analytics выключен или Group by по кастомному полю недоступен

**Симптом:** ADO не даёт выбрать `Custom.FoundStage` в Chart for Work Items.

**Действия:**

1. Включить Analytics по инструкции в [ado-dashboard-setup.md](ado-dashboard-setup.md).
2. Временная замена: **Query Results** с сортировкой по ID + колонка FoundStage (ручной просмотр выборки).
3. Для тренда: использовать встроенный **CFD по доске** (если команда ведёт спринты на той же доске), не смешивая с WIQL-метриками без пояснения.

---

## 6. Запросы из скрипта ещё не созданы (мало прав)

**Симптом:** нет `QA Bugs By State This Month` и т.п.

**Действия:**

1. Собрать MVP только из **ручных WIQL** из [ado-dashboard-setup.md](ado-dashboard-setup.md) (строки 1–3 + found_stage при наличии поля).
2. После `create_ado_queries.py` — сверить имена и добавить chart-строку по [dashboard-widget-map.md](dashboard-widget-map.md).

---

## 7. Порог эскалации (рекомендация для регламента)

| Сигнал | Действие |
|--------|----------|
| Prod bugs (MTD или 30d) растёт 2 периода подряд | Вынести на risk review |
| Legacy backlog растёт при стабильном inflow | Проверить приоритизацию и WIP лимиты |
| > X% закрытых Feature без QaDecision | Стоп на «закрытие без sign-off» до выравнивания |

Значения X зафиксировать с QA lead (не в коде).
