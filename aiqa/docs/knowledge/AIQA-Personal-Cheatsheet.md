# 🤖 AI QA Framework — Личная шпаргалка

> Что это: набор AI-скиллов для ежедневной QA-работы в ETNA_TRADER.
> Где лежит: репозиторий DevReps, папка `aiqa/`
> Дата: 2026-05

---

## Главное в трёх строках

Есть набор команд (`/qa`, `/ai-settings`, `/sr` и другие), которые запускаются в Cursor или Claude и генерируют реальные артефакты: тест-планы, тест-кейсы, release notes, code review, RCA-отчёты. Не нужно каждый раз объяснять контекст — скиллы уже знают как работать с ETNA_TRADER.

**Новый скилл `/learn`** — запускается в конце каждой сессии. Фиксирует что нашёл: опасные зоны кода, скрытые зависимости, пробелы в тестах. Генерирует готовый YAML для добавления в impact-map и repo-index. Контекст больше не теряется.

---

## 🚀 Как начать прямо сейчас

Открыть в Cursor или Claude и написать:

```
/qa [название фичи]
```

Всё. Агент сам разберётся что делать дальше.

### Свободные промпты (без dedicated команды)

**Impact Analysis** — когда нужно понять blast radius изменений:
```
Прочитай aiqa/impact-map.yaml.
Какие правила совпадают с этими путями: [список путей]?
Какие required_checks применяются? Какие репозитории затронуты?
```

**RCA** — разбор инцидента:
```
Проанализируй логи в [путь] и SQL результаты в [путь].
Время инцидента: [дата/время].
Построй дерево гипотез. Ранжируй по доказательствам за/против.
Определи наиболее вероятный root cause.
Сохрани: tasks/[папка-инцидента]/rca-report.md
```

## 🔄 Типичные сценарии

### QA-инженер — новая задача

```
1. Получил фичу и тех-декомпозицию из tasks/task-[дата]-[фича]/
2. /qa [название фичи]
   → получаю test-plan-[фича].md

   → получаю test-cases-[фича].md
4. После прогонов: /qa → "coverage review"
   → получаю coverage-review-[фича].md
```

### Разработчик — перед PR

```
1. /ai-settings PRE_COMMIT_CHECK   → проверка всего
2. /ai-settings RELEASE_NOTES      → release notes из diff
3. /sr [путь к задаче]             → code review
```

### Тимлид — новая фича

```
1. /nf [описание фичи]   → скоупинг
2. /ct [фича]            → декомпозиция
3. QA запускает /qa      → тест-план
```

### Инцидент

```
1. Сложил логи + SQL в tasks/[инцидент]/
2. Запустил RCA промпт
3. Проверил rca-report.md
```

### После PR — обновить канон

```
1. /pr-enrich [номер PR]    → pr-enrichment.md с YAML предложениями
2. Проверил, скопировал нужное в aiqa/impact-map.yaml или repo-index.yaml
3. Добавил запись в knowledge-journal.md
4. Закоммитил
```
Работает без локального клона — можно вставить список файлов из GitHub вручную.

## 💾 /learn — как не терять контекст сессии

### Зачем

Пока решаешь задачу в Cursor — находишь неожиданное: опасный участок кода, скрытую зависимость между сервисами, пробел в тестах, непонятное поведение домена. Закрываешь сессию — всё это исчезает. Через месяц сам (или коллега) наступает на то же самое.

`/learn` — это 2 минуты в конце сессии которые экономят часы потом.

### Как работает

```
1. Запускаешь /learn в конце задачи
2. Скилл смотрит git diff, читает task.yaml — понимает контекст
3. Задаёт 5 вопросов:
   • Что удивило в коде?
   • Что помечать как hotspot (опасная зона)?
   • Что не покрыто тестами?
   • Что узнал о доменной логике?
   • Что осталось непонятным?
4. Сохраняет discoveries.md в папку задачи
5. Генерирует готовый YAML для вставки в impact-map или repo-index
```

### Что получаешь

```
tasks/task-2026-05-12-margin-call/
  discoveries.md       ← структурированные находки
```

Внутри:
- Описание каждой находки с типом и доказательством
- Готовый YAML для копирования в `aiqa/impact-map.yaml` (новые правила)
- Готовый YAML для копирования в `aiqa/repo-index.yaml` (hotspots)

### Промоция (когда готов)

```
1. Открой discoveries.md
2. Возьми YAML-сниппет из раздела "Кандидаты на промоцию"
3. Вставь в aiqa/impact-map.yaml или aiqa/repo-index.yaml
4. Обнови статус в discoveries.md: draft → promoted
5. Добавь запись в aiqa/docs/knowledge/knowledge-journal.md
6. Закоммить
```

Делать сразу необязательно. discoveries.md остаётся в git — можно вернуться потом.

### Типы находок

| Тип | Что значит | Куда промотировать |
|---|---|---|
| `hotspot` | Опасная зона — проверять при каждом касании | repo-index.yaml |
| `impact_rule` | "Если меняется X → проверь Y" | impact-map.yaml |
| `domain_pattern` | Правило бизнес-логики, неочевидное поведение | остаётся в discoveries.md |
| `test_gap` | Не покрыто тестами | остаётся, добавляется в backlog |
| `open_question` | Непонятно, нужно исследовать | остаётся |

## 🔗 /pr-enrich — обогащение индекса из PR

### Зачем

Impact Map и Repo Index не обновляются сами. После каждого PR в репозиторий попадают новые пути, новые сервисы, новые зависимости — которых нет в `aiqa/impact-map.yaml` и `aiqa/repo-index.yaml`. Через месяц кто-то меняет тот же файл — и агент не знает что нужно проверить, потому что правила не покрывают этот путь.

`/pr-enrich` — смотрит на PR, сравнивает с тем что есть в YAML-файлах, и предлагает что добавить.

**Главный бонус:** работает даже без локального клона репозитория. Достаточно вставить список файлов из GitHub.

### Как запустить

**Вариант 1 — есть локальный клон:**
```
/pr-enrich 123
```
Скилл сам возьмёт `git diff origin/main...HEAD`.

**Вариант 2 — нет клона (открыл PR на GitHub):**
```
/pr-enrich

[вставь сюда список файлов из вкладки "Files changed" на GitHub]
```

### Что получаешь

```
tasks/pr-enrich-123/
  pr-enrichment.md   ← анализ + YAML для копирования
```

Внутри:
- Таблица: какие пути покрыты правилами, какие нет
- Конкретные gap'ы: `path_not_covered`, `domain_missing`, `hotspot_candidate`
- Готовый YAML для вставки в `aiqa/impact-map.yaml` и `aiqa/repo-index.yaml`
- Все предложения имеют `confidence: low` — нужна ручная проверка перед промоцией

### Промоция

```
1. Открой tasks/pr-enrich-[число]/pr-enrichment.md
2. Проверь предложенный YAML — он правдоподобен?
3. Вставь в aiqa/impact-map.yaml или aiqa/repo-index.yaml
4. Добавь запись в aiqa/docs/knowledge/knowledge-journal.md
5. Закоммить
```

Скилл **никогда не редактирует** YAML-файлы сам. Только предлагает — ты решаешь.

## 🔑 Ключевые понятия за 2 минуты

**Skill** — файл с инструкцией для AI-агента. Описывает: что принять на вход, что отдать на выход, какие правила соблюдать. Вызывается через `/команду`.

**Agent** — AI, который запускает один или несколько скиллов. Например, `/sr` запускает двух агентов параллельно: архитектурного ревьюера и security-ревьюера.

**Impact Map** — таблица: «если изменён файл по такому-то пути → нужно проверить вот это». Помогает не забыть про смежные риски.

**Repo Index** — список репозиториев с описанием доменов, сервисов, hotspots. Агенты используют его чтобы понимать контекст изменений.

**Task Carrier** *(планируется)* — YAML-файл задачи, который накапливает контекст от лида → дева → QA и передаёт его агенту одним файлом.
