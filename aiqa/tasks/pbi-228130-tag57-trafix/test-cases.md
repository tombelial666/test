# Тест-кейсы (конкретные сценарии)

Формат: **ID — предусловия — шаги — ожидаемый результат — как проверить**.

Префикс **TC-228130-** для трассировки.

---

## TC-228130-01 — Дробный объём, маршрут Trafix

- **Предусловия:** Символ с дробными долями; маршрут **Trafix** (или RQD, если подтверждено, что FIX идёт через тот же конвертер); ордер **не** notional (`CashAmountOrder` не выставлен); количество с дробной частью (например 11.5).
- **Шаги:** Отправить новый рыночный/лимитный ордер с дробным qty.
- **Ожидание:** В **Standard Header** tag **57** = **`FRAC`**. В теле — согласно существующей логике: при необходимости **20038** = фактическое qty, **38** = округление вверх до целого (как в тесте `SingleLeg_SecurityType_CommonStock_FractionalQuantity`: ожидание 38=12 для 11.5).
- **Проверка:** FIX capture / лог с разбором тегов; сравнить с `OrderTestData_Trafix.json` на ветке.

---

## TC-228130-02 — Целый объём, Trafix

- **Предусловия:** Тот же маршрут; количество целое (например 11.0 или 11).
- **Шаги:** Новый ордер.
- **Ожидание:** **57** не содержит `FRAC` (tag 57 отсутствует или пусто — как до фичи для таких ордеров). Нет ложной дробной ветки.
- **Проверка:** Outbound FIX; отсутствие `57=FRAC` в header.

---

## TC-228130-03 — Notional (cash amount), Trafix

- **Предусловия:** Ордер с флагом **CashAmountOrder** (`ExecInst` включает бит как в системе — в тестах значение **1**); сумма в поле цены (например 1000 USD).
- **Шаги:** Отправить notional ордер.
- **Ожидание:** **Tag 152** = сумма ордера (cash); **38** не отправляется / null; **57** не `FRAC` (логика `IsFractional` ложна для notional).
- **Проверка:** FIX: наличие 152, отсутствие 38; header без 57=FRAC.

---

## TC-228130-04 — Notional + дробное поле quantity (если продукт позволяет)

- **Предусловия:** Как в юнит-тесте `SingleLeg_SecurityType_CommonStock_CashAmountOrder_Fractional`: `ExecInst=1`, `Price=1000`, `Quantity=11.5`.
- **Шаги:** Создать такой ордер.
- **Ожидание:** Приоритет notional: **152** = 1000; **38** = null; **57** не FRAC (даже при дробном qty в модели).
- **Проверка:** Сверка с JSON-override в `OrderTestData_Trafix.json`; FIX на стенде.

---

## TC-228130-05 — Не Trafix / не RQD (контроль маршрута)

- **Предусловия:** Другой execution venue, тот же тип ордера (дробный share).
- **Шаги:** Отправить дробный ордер на другой маршрут.
- **Ожидание:** Поведение **как в базовой линии** для этого конвертера; **не** ожидать автоматически `FRAC` в 57 из этого PR (изменён только Trafix).
- **Проверка:** Лог маршрута + FIX другого коннектора; регресс «не сломали остальное».

---

## TC-228130-06 — Конфликт/невалидные условия

- **Предусловия:** Сценарии, которые продукт может считать невалидными: notional без суммы, нулевой qty, отключённый fractional на символе.
- **Шаги:** Попытаться отправить; проверить отказ OMS **до** FIX.
- **Ожидание:** Корректная ошибка на стороне платформы; **не** отправлять некорректный FIX с противоречивыми 38/152/57.
- **Проверка:** Сообщения об ошибке в UI/API; отсутствие «полу-собранного» FIX.

---

## TC-228130-07 — Регрессия обычных потоков (не цель изменения)

- **Предусловия:** Trafix; целочисленный ордер; short / held / retail (если есть в проде).
- **Шаги:** Прогнать смоук из существующих кейсов в `OrderTestData_Trafix.json` (ETF, SellShort, Held, BuyToCover и т.д.).
- **Ожидание:** Совпадение с ожидаемыми тегами в JSON (кроме явно новых полей в базовом Expected, если ветка добавила tag 0 / 20038 / 152 в шаблон).
- **Проверка:** Автотесты + выборочно FIX.

---

## Минимальный набор для регресса автотестов

Класс: **`OrderConverterTest`** (`Etna.Trading.ExecutionVenues.Tests`), источник данных — glob `CommonTests/OrderTestData_*.json` (в т.ч. `OrderTestData_Trafix.json`).

Пример (из корня solution, уточнить путь к `.csproj`):

```text
dotnet test src/Etna.Trading.Components/Etna.Trading.ExecutionVenues.Tests/Etna.Trading.ExecutionVenues.Tests.csproj --filter "FullyQualifiedName~OrderConverterTest"
```
