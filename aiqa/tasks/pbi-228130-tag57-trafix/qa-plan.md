# План QA: Tag 57 / 152, Trafix

## Область (scope)

- Исходящие FIX-сообщения для маршрута **Trafix** (`TrafixOrderConverter`).
- Типы ордеров: **дробное количество (share)**, **целое количество**, **notional (CashAmountOrder)** и **комбинация notional + дробное количество в поле** (как в unit-тесте — проверить соответствие доменной модели).

## Окружения и зависимости

- **Стенд** с реальной маршрутизацией в Trafix (или тестовый FIX-приёмник), где виден **сырой FIX** или лог парсера.
- Аккаунт/символ с **дробными** лотами и с **notional**, если включено продуктом.
- Доступ к **логам FIX-сессии** (gateway / OMS) или **pcap** / export сообщений — для объективной верификации тегов.
- Юнит-тесты: проект `Etna.Trading.ExecutionVenues.Tests`, данные `OrderTestData_Trafix.json` (после мержа ветки).

## Требуемые доказательства (evidence)

1. **Outbound message** для нового ордера: наличие/отсутствие **tag 57** в **Standard Header** (в тестах это часто сериализуется как вложенная группа / tag 0 с `57: FRAC`).
2. Для notional: **tag 152** = сумма в валюте (в коде — `order.Price`), **tag 38** отсутствует или не заполняется (как в тестовых ожиданиях).
3. Для дробного share: при необходимости **20038** и **38** (как в существующей логике) + **57 = FRAC**.

## Подход к тестированию

1. **Автоматически:** прогон тестов с `OrderTestData_Trafix.json` на ветке `228130-Tag-57-for-Trafix`.
2. **Интеграционно:** 1–2 ордера каждого класса на стенде с **захватом FIX-сообщения** до ухода на биржу/брокера.
3. **Сравнение:** один и тот же сценарий на **другом** маршруте (не Trafix) — убедиться, что изменения не «протекли» (ожидаемо не должны).

## Разрешение маршрута vs заполнение тегов

- **Факт из кода:** `IsFractional` и ветка `CashAmountOrder` используют **объект `IOrder`** до/независимо от финальной сборки header; **маршрут** должен быть выбран так, чтобы вызвался именно `TrafixOrderConverter`. Иначе проверка tag 57 бессмысленна.
- Проверить в **логах маршрутизации** (какой execution venue / converter), что заявка пошла через Trafix.

## Логирование и верификация сообщений

- Искать в логах **полный FIX text** или hex-дамп для сообщения типа **D** (NewOrderSingle).
- Проверить вручную: **Tag 57** в header; **Tag 152** в теле для notional.
- Если доступен только бизнес-лог без FIX — **недостаточно** для строгой проверки тегов; минимум — парсированный список тегов из интеграционного слоя.

## Пересечение fractional и notional

- **По коду:** если установлен `CashAmountOrder`, `IsFractional` возвращает **false**, поэтому **57 = FRAC не ставится**, даже если в модели quantity дробное.
- **Инференс:** доменно «notional + дробный quantity» может быть редким или запрещённым; тест в JSON добавляет сценарий — нужно прогнать и при необходимости уточнить у аналитика допустимость.
