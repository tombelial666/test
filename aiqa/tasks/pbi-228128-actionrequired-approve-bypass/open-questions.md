# Открытые вопросы (блокируют полную уверенность)

1. **Обход для всех причин `ActionRequired`**  
   Условие в PR завязано только на пару **Approve + исходный статус ActionRequired**, без проверки **причины** (Trulioo vs ошибка БД vs другое). Нужно ли ограничить обход только сценарием «после failed Trulioo», и как это выразить в коде или политике?

2. **Соответствие PR 15452 и локального `git diff`**  
   Номер PR в git не хранится. Нужна ручная сверка в Azure DevOps: тот же merge-base, нет дополнительных коммитов относительно `origin/dev` в вашем clone.
 
3. **Доменный переход `ActionRequired → Approved`**  
   Явно ли задокументирован и покрыт тестами в домене `AccountRequest`? При сомнениях — интеграционный тест на успешный `Approve` из `ActionRequired`.

4. **Комплаенс и аудит**  
   Достаточно ли для регуляторных требований записи в истории заявки + идентификация оператора в `Approve`, или нужен отдельный audit event «verification bypassed»?

5. **Параллельные API**  
   Есть ли альтернативные endpoints/админки, меняющие статус без `ModifyAccountRequest` — и требуют ли они того же правила?
