# Влияние и регрессии

## Прямо затронутая логика

- **Единственный вход:** `ModifyAccountRequestHandler.Handle` — ветка обработки действия **Approve** и последующий блок проверки статуса `Approved` с условным вызовом **Trulioo** (`AccountRequestVerify`).
- **Поведение:** при переходе **Approve**, когда **до** обработки команды статус был **ActionRequired**, повторный вызов `AccountRequestVerify` **не выполняется**.

## Косвенное влияние

- **Очередь на clearing / фоновая обработка:** если раньше заявка «застревала» в `Approved` с неуспешной повторной верификацией, после PR при сценарии «Approve из ActionRequired» верификация не блокирует — **ProcessAccountRequest** / job могут выполниться при прочих равных (как задумано для ручного доверия).
- **Аудит и комплаенс:** обход автоматической identity-проверки увеличивает вес **ручного решения** и записей в истории заявки (`Approve` + комментарий).

## Риск общего кода (shared path)

- Изменения **локализованы** в одном handler; другие обработчики не затронуты **этим** diff.
- **Риск дрейфа:** если появится второй путь «одобрить из ActionRequired» без `ModifyAccountRequest`, поведение может разойтись.

## Матрица регрессии (ожидания)

| Сценарий | Статус до команды | Действие | Вызов `AccountRequestVerify` (если mandatory + open) |
|----------|-------------------|----------|------------------------------------------------------|
| A — стандартный approve | `Submitted` (типично) | `Approve` | **Да** (как baseline) |
| B — цель PR | `ActionRequired` | `Approve` | **Нет** (обход) |
| C | `ActionRequired` | `Reject` / `Cancel` | Не применимо к ветке `Approved` + verify |
| D | `Submitted` | `Submit` / др. | Без изменений в части verify-after-approve |

## Скрытые риски

- **Любая** причина статуса `ActionRequired` (не только Trulioo): при **Approve** верификация также **не** вызывается — продукт должен явно принять или сузить условие (например, только если последний перевод в ActionRequired был из verification).
- **Отсутствие автотестов в PR:** регресс при рефакторинге условия выше по файлу.
- **Домен `CheckValidStateTransition`:** в коде есть технический долг (ранний `return` / недостижимые ветки); опора только на чтение без интеграционного теста рискованна.

## Что перепроверить вне «ровно нового» поведения

- **Approve** из `Submitted` с обязательной Trulioo — регресс «всё ещё вызывается verify».
- **SyncProcessing = true** и **false** — оба пути сохранения и enqueue job.
- Логи AMS на предмет **отсутствия** повторного outbound Trulioo при сценарии B.
