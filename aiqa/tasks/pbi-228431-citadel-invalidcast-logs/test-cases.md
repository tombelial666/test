## TC-228431-LOG-01 — Ретро: полный след инцидента в логах

**Цель:** доказать по логам цепочку «исходящий replace → входящий ER → ошибка конвертера» для **одной** заявки.

### Предусловия

- Известны: время инцидента (UTC), **ClOrdID** replace-ноги, при возможности **37** (OrderID), символ.
- Есть доступ к логам за эту дату.

### Шаги

1. Открыть Kibana → Discover, выставить интервал **T0 − 3 min … T0 + 5 min** (UTC), `T0` — время первого `InvalidCastException` или ER.
2. Поиск по одному из: `InvalidCastException`, `MapFromMessageExecution`, уникальный фрагмент `ClOrdID`, `2026031702KC2` (пример 37) — пока не останется **одна** заявка в выборке.
3. Найти запись `[ERR]` с `Error accrued during processing event` и subject, содержащим `mdb_citadel_equities` (или ваш venue id).
4. Вверх по времени на том же хосте/потоке: найти **входящий** `Fix Message Received` / `Message Received` с `35=8` или `Type: 8`.
5. Ещё выше: **исходящий** `Send Fix Message` с `35=G` и `41=` на старый `ClOrdID`.
6. Вырезать **20–40 строк до** ERR и **10 после**: сырой FIX `35=8`, полный stack до `CitadelEqOrderConvertor.cs:…`.

### Ожидаемый результат

- Видна последовательность: **G → 8 → ERR**; в stack есть `MapFromMessageExecution`.

### Артефакт

- Текстовый файл или скрин; в тикете: UTC, инстанс, версия, маскированные id.

## TC-228431-LOG-06 — Диагностика «один раз на тип» после рестарта OMS: **размещение (Single placed)**

**Цель:** проверить поведение, о котором договорились с разработкой: диагностическое сообщение **появляется один раз на тип** после старта OMS; **повтор** того же типа **без рестарта** не повторяет лог.

### Предусловия

- Есть права **перезапустить OMS** на тестируемой среде (или согласовано окно с DevOps).
- Известны **точные строки** диагностики для сценария «новая заявка» (от dev). Если нет — использовать как минимум фильтр `CitadelEqOrderConvertor` + уровень из PR.

### Шаги

1. Зафиксировать время **T_restart** (UTC) — момент рестарта OMS или время «готовности» сервиса после старта.
2. Выставить **новую** заявку (обычное размещение, **не** replace) на нужном **venue** (например Citadel).
3. В Elastic: интервал **T_restart … T_restart + 10 min**, поиск диагностической строки для этого типа действия.
4. **Ожидание шага 3:** запись **найдена** (первое срабатывание после рестарта).
5. **Не перезапуская OMS**, выставить **вторую** заявку того же типа на том же venue (аналогичные параметры).
6. В Elastic: то же окно, расширить до **T_restart + 30 min**, тот же поисковый запрос.
7. **Ожидание шага 6:** **вторая** диагностическая запись такого же типа **отсутствует** (или дубли не ожидаются по правилам от dev — уточнить при расхождении).

### Ожидаемый результат

- Таблица:

  | Шаг | Действие | Диагностический лог найден? | Timestamp UTC |
  |-----|----------|-----------------------------|---------------|
  | 1 | Рестарт OMS | — | T_restart |
  | 2 | Первая заявка (place) | Да | |
  | 3 | Вторая заявка (place) | Нет | — |

### Артефакт

- Ссылка на Discover или экспорт **двух** запросов (после 1-й и после 2-й заявки).

## TC-228431-LOG-09 — «Один раз на **venue**»

**Цель:** по пояснению разработки: то же правило «один раз» может применяться **отдельно для каждого venue**.

### Предусловия

- Доступны **два разных venue** (две биржи/маршрута) с тем же типом действия, например **place**.

### Шаги

1. **Рестарт OMS**, **T_restart**.
2. Выполнить **place** на **venue A**; проверить диагностику — **должна быть** (первый раз для A после рестарта).
3. **Без рестарта** выполнить **place** на **venue B** (тот же тип действия).
4. **Ожидание:** диагностика для **venue B** снова **один раз** (как новый ключ), даже если для A уже был лог.
5. **Без рестарта** выполнить **второй place** на **venue A** и **второй** на **venue B**.
6. **Ожидание:** для каждого venue **повторной** диагностики **нет** (до следующего рестарта).

### Ожидаемый результат

- Таблица: venue | первое действие после рестарта | лог да/нет | второе действие | лог да/нет.

### Если второй venue недоступен

- Записать: «TC-LOG-09 не выполнялся: второй venue не настроен».
