# test-cases — Leaderboard (smoke + регресс)

**Префикс трассировки:** `TC-LB-`  
**Уровень зрелости (aiqa):** review-grade — чеклист для ручного прогона и ревью; автоматизация по кейсам не заявлена.  
**Стенд (пример):** `https://etna-demo-ci-int-2.etnasoft.us` / `https://pub-api-etna-demo-ci-int-2.etnasoft.us` — подставить актуальный целевой URL.

**Секреты:** пароли, cookies, `.ASPXAUTH`, заголовки с учётными данными в репозиторий **не** коммитить; в прогоне — тестовая учётка и локальные переменные окружения.

---

## 0. Объект проверки и точки входа

| Канал | Назначение |
|--------|------------|
| Веб UI | Экран Leaderboard: фильтры, таблица, пагинация, сортировка, Export, Auto Refresh, колонки. |
| Веб API | `GET /public/v1/accounts-with-balances` с cookie-сессией и `et-app-key` (same-origin). |
| Pub API | `POST /api/token` → `GET /api/v1/accounts-with-balances` с Bearer (другой ключ приложения). |

**Общие query-параметры (как в продукте):** `pageNumber`, `pageSize`, `isAscending`, `propertyName`, `filter`.

## 1. Smoke (блокирующие)

### TC-LB-01 — Открытие Leaderboard под авторизованным пользователем

- **Предусловия:** Пользователь с правом видеть Leaderboard; стенд доступен.
- **Шаги:** Войти → открыть Leaderboard.
- **Ожидание:** Страница загружается без необработанной ошибки в консоли; видны заголовок, фильтры (Id, Clearing Account, Rep Code), кнопки Filter / Reset / Export to CSV, таблица с колонками (Rank, Id, Clearing Account, Created Date, Margin Type, Cash, Change, Change %, Account Value, Excess, Net Liq Value, показатели P/L, Rep Code, ROI, Login — по фактическому макету стенда).
- **Проверка:** Визуально + DevTools Console без критичных ошибок.

### TC-LB-02 — Первый запрос списка (API или перехват из UI)

- **Предусловия:** Сессия валидна.
- **Шаги:** Вызвать `accounts-with-balances` с параметрами по умолчанию (как при первом открытии) **или** снять ответ из Network.
- **Ожидание:** HTTP `200`; тело JSON с массивом `Result`, полем `TotalCount`, при необходимости `NextPageLink` / `PreviousPageLink`.
- **Проверка:** Статус, парсинг JSON, наличие перечисленных полей верхнего уровня.

## 2. Целостность пагинации и ответа (обязательная регрессия)

### TC-LB-03 — Согласованность страницы, размера и числа строк в `Result`

- **Предусловия:** Известны `pageNumber`, `pageSize`, `TotalCount` из ответа.
- **Шаги:** Зафиксировать один полный ответ API для выбранной страницы (не обрезанный вручную). Пусть `N = Result.length`.
- **Ожидание:** Число `N` **согласовано** с правилами пагинации продукта, например:
  - для страниц **до** последней: `N === pageSize` (если данных хватает);
  - для **последней** страницы: `N === TotalCount - pageSize * (pageNumber - 1)` при **1-based** нумерации страниц **или** эквивалент по документированной формуле **0-based**.
- **Проверка:** Явное сравнение; при несоответствии — **дефект** или уточнение семантики `pageNumber`/`TotalCount` у разработки (не игнорировать, см. наблюдение в §9).

### TC-LB-04 — Ссылки `NextPageLink` / `PreviousPageLink`

- **Шаги:** На первой странице, средней, последней — проверить наличие/пустоту ссылок.
- **Ожидание:** На первой странице нет перехода «назад» или он пустой по правилам API; на последней нет «вперёд»; на средних есть оба при необходимости. URL содержат те же `pageSize`, `propertyName`, `isAscending`, `filter`, что и текущий запрос.
- **Проверка:** Сравнение query string.

### TC-LB-05 — UI: число строк таблицы = длине `Result` для того же запроса

- **Шаги:** Сопоставить последний запрос `accounts-with-balances` в Network с отображёнными строками (без скрытых фильтров, если таблица не виртуализует пустые слоты ошибочно).
- **Ожидание:** Количество видимых строк данных = `Result.length` для этого ответа.
- **Проверка:** Счёт строк vs JSON.

## 3. Сортировка

### TC-LB-06 — Сортировка по Change % по возрастанию

- **Шаги:** `propertyName=changePercent`, `isAscending=true`; пройти по нескольким страницам или сравнить соседние строки.
- **Ожидание:** Порядок значений Change % неубывающий (с учётом округления в UI).
- **Проверка:** Значения из API (`changePercent` в `BalanceAttributes` или согласованное поле) и из колонки UI.

### TC-LB-07 — Сортировка по Change % по убыванию

- **Шаги:** `isAscending=false`.
- **Ожидание:** Невозрастающий порядок.
- **Проверка:** Аналогично TC-LB-06.

### TC-LB-08 — Смена сортируемой колонки (регресс по кликабельным заголовкам)

- **Шаги:** По очереди включить сортировку по другим доступным числовым/строковым колонкам из UI (как в продукте).
- **Ожидание:** Запрос уходит с корректным `propertyName`; порядок соответствует выбранному направлению; индикатор сортировки на заголовке верный.
- **Проверка:** Network + визуально.

## 4. Фильтры

### TC-LB-09 — Фильтр по Id

- **Шаги:** Ввести известный Id из текущего набора; Filter.
- **Ожидание:** Таблица показывает только подходящие строки; `TotalCount` отражает отфильтрованный набор; параметр `filter` в запросе соответствует продукту.
- **Проверка:** UI + API.

### TC-LB-10 — Фильтр по Clearing Account

- **Шаги:** Аналогично TC-LB-09 для Clearing Account.
- **Ожидание:** Узкий набор; нет «лишних» строк.
- **Проверка:** Сверка поля Clearing Account в каждой строке.

### TC-LB-11 — Фильтр по Rep Code

- **Шаги:** Аналогично для Rep Code.
- **Ожидание:** Все строки с выбранным Rep Code (или пусто, если нет совпадений — сообщение/пустая таблица по правилам UI).
- **Проверка:** UI + при необходимости ответ API.

### TC-LB-12 — Reset

- **Предусловия:** Применён хотя бы один фильтр.
- **Шаги:** Reset.
- **Ожидание:** Поля фильтров очищены; данные как при первом открытии; запрос без ограничивающего фильтра (или с пустым `filter` — как в продукте).
- **Проверка:** Network + UI.

## 5. Действия грида

### TC-LB-13 — Export to CSV

- **Шаги:** При текущих фильтре и сортировке нажать Export to CSV.
- **Ожидание:** Файл скачивается; кодировка и разделитель читаемы; заголовки соответствуют видимым/выбранным колонкам; данные согласованы с экраном для **экспортируемого** диапазона (вся выборка или текущая страница — **зафиксировать ожидание по продукту** и проверить).
- **Проверка:** Открыть CSV; выборочно сверить строки с UI/API.

### TC-LB-14 — Auto Refresh

- **Шаги:** Включить Auto Refresh; наблюдать 2–3 цикла (или интервал из кода/доки).
- **Ожидание:** Периодические запросы к `accounts-with-balances`; нет дублирования строк; при неизменных данных — допустимое обновление без мигания; при ошибке сети — предсказуемое поведение (не молчаливый пустой грид без индикации, если так задумано).
- **Проверка:** Network (интервал), Console.

### TC-LB-15 — Настройка колонок (Change Columns)

- **Шаги:** Скрыть/показать колонки через ссылку внизу грида.
- **Ожидание:** Таблица обновляется; состояние сохраняется в рамках сессии/профиля по правилам приложения; экспорт учитывает видимые колонки или документированное правило.
- **Проверка:** Визуально + при необходимости CSV.

## 6. Согласованность данных: UI ↔ API

### TC-LB-16 — Строка-в-строку для выбранного счёта

- **Шаги:** Выбрать строку; для того же `Id` / `ClearingAccount` сравнить: Cash, Change, Change %, Account Value, Excess, Net Liq Value, P/L поля, Rep Code, ROI, Login, Created Date, Margin Type.
- **Ожидание:** Значения в UI совпадают с источником в JSON (корневые поля и/или `BalanceAttributes` по имени: `cash`, `changeAbsolute`, `changePercent`, `equityTotal`, `excess`, `netLiquidity`, `openPL`, `openPLDay`, `openPLPercent`, `totalPL`, `ROI` и т.д.) с учётом округления и локали даты/числа.
- **Проверка:** Таблица соответствия колонка → поле JSON задокументирована в заметках прогона.

### TC-LB-17 — Ранг (Rank)

- **Шаги:** Сравнить отображаемый ранг (включая составной формат, если есть) с полем `Rank` и `TotalCount` в ответе.
- **Ожидание:** Правило отображения **однозначно** и совпадает с продуктом (если расхождение — завести уточнение у разработки, см. `task.yaml` unknowns).
- **Проверка:** Запись наблюдения в отчёт прогона.

### TC-LB-18 — Расхождения `BalanceAttributes` и корня

- **Шаги:** Для счёта, где `equityTotal` (атрибут) и `EquityTotal` (корень) или аналоги различаются, зафиксировать, какое значение показывает колонка Account Value.
- **Ожидание:** Поведение стабильно и совпадает с документацией/кодом; нет «прыжка» между релизами без изменения требований.
- **Проверка:** Снимок JSON + скрин колонки.

## 7. Два канала: веб-сессия и pub-api

### TC-LB-19 — Pub API: получение токена

- **Шаги:** `POST /api/token` с валидными `username`, `password`, `et-app-key` (только на тестовом стенде).
- **Ожидание:** Успешный ответ с токеном для последующих запросов.
- **Проверка:** Статус 200 (или ожидаемый код продукта); наличие токена в теле/заголовках по контракту.

### TC-LB-20 — Сравнение веб `public/v1` и pub-api `api/v1` при одинаковых параметрах

- **Предусловия:** Токен pub-api получен; cookie-сессия веб активна.
- **Шаги:** Одинаковые `pageNumber`, `pageSize`, `propertyName`, `isAscending`, `filter` на обоих хостах.
- **Ожидание:** `TotalCount` совпадает; порядок `Id` в `Result` совпадает; ключевые поля по одному и тому же счёту совпадают **или** задокументированное допустимое отличие (версия API, маскирование).
- **Проверка:** Дифф JSON (допускается скрипт сравнения в заметках прогона).

## 8. Негатив и устойчивость

### TC-LB-21 — Неавторизованный / просроченная сессия

- **Шаги:** Открыть Leaderboard без логина или с истёкшими cookies / после logout.
- **Ожидание:** Редирект на логин или 401/403 для API; нет «пустой таблицы без объяснения», если так не принято в продукте.
- **Проверка:** Статус и UX.

### TC-LB-22 — Некорректные параметры пагинации

- **Шаги:** `pageNumber=0`, отрицательный, заведомо больше максимума; `pageSize` граничные значения (0, 1, очень большое — по допустимости API).
- **Ожидание:** Предсказуемая валидация (400) или безопасное приведение к дефолту — **как в контракте**; без 500 и утечки стека в клиент.
- **Проверка:** Статусы и тело ошибки.

### TC-LB-23 — Ошибка сети / таймаут при Auto Refresh

- **Шаги:** Симуляция offline или блокировка URL в DevTools после включения Auto Refresh.
- **Ожидание:** Повторы по политике; сообщение пользователю или индикатор ошибки; приложение не зависает.
- **Проверка:** Console / UI.

## 9. Нефункциональные

### TC-LB-24 — Время ответа API

- **Шаги:** Типичный запрос первой страницы и страницы с большим `pageSize` (если разрешено).
- **Ожидание:** В пределах согласованного SLA стенда; нет многократного превышения без причины (нагрузка на стенд учитывается).
- **Проверка:** Timing в Network.

### TC-LB-25 — Консоль браузера при сценариях TC-LB-01–15

- **Ожидание:** Нет необработанных исключений, влияющих на функциональность.
- **Проверка:** Console.

## 10. Явная проверка по наблюдению со стенда (без додумываний)

### TC-LB-26 — Зафиксированное расхождение метаданных и тела ответа

- **Контекст:** На сохранённом примере ответа: `TotalCount: 664`, пагинация указывает на последнюю страницу (`PreviousPageLink` на стр. 66, `NextPageLink` пустой), при этом в `Result` **один** элемент, на UI — **одна** строка (полный кадр страницы).

- **Шаги:** Повторить запрос на том же стенде с теми же параметрами; сохранить **полный** raw response и HAR.

- **Ожидание (регресс):** Либо `Result.length` становится согласованным с `TotalCount`/`pageNumber`/`pageSize` по документированной формуле, либо дефект/документация обновлена с объяснением семантики.

- **Проверка:** Запись в отчёт: фактическое `N`, формула ожидания, вывод «OK / дефект / ожидаемо по спецификации».
