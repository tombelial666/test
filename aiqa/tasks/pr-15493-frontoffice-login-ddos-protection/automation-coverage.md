# Покрытие PR 15493: автотесты vs ручные / другие проверки

**Код автотестов:** `qa/frontoffice_login_guard/test_frontoffice_login_guard.py`  
**Спека:** [`frontoffice-login-guard-full-spec.md`](frontoffice-login-guard-full-spec.md)

Документ фиксирует, **что именно** проверяет pytest-набор и **что намеренно или по природе сценария** остаётся вне этого файла (ручной регресс, SQL trace, другой контур продукта).

---

## 1. Что проверено автотестами (интеграция на стенде)

| Область | Содержание проверки |
|--------|----------------------|
| **MVC LogOn, первая попытка** | Успешный POST с валидными кредами: нет 429 «на входе», коды ответа в ожидаемом наборе (TC-01 по HTTP). |
| **ExternalLogOn, контракт** | Эндпоинт отвечает, JSON парсится при негативном теле; при тех же кредах, что и LogOn — при успехе есть `redirectUrl` (TC-02). |
| **Rate limit** | После серии POST с **одним** логином и неверным паролем приходит **HTTP 429** и заголовок **Retry-After** (TC-03; ветка лимита **per IP+login** 5/60 согласно коду). |
| **External без per-IP-only** | Несколько **разных** логинов с одного IP без 429 «только из-за общего per-IP» (TC-04). |
| **Bloom / precheck, грубо** | Заведомо случайный логин: нет 500 и нет 429 на одну попытку; при желании строго **401** — переменная `FO_STRICT_BLOOM_401=1` (TC-05). |
| **Дым страницы логона (опционально `FO_GUARD07_*`)** | Не приёмка PR: только проверка, что GET LogOn живой. **TC-FO-GUARD-07 (Regal, переключение контекста) PR 15493 не решает** — см. §2.3. |
| **«Холодный» сценарий** | GET LogOn + POST с подставным логином: нет 500 (TC-08 в упрощённом виде). |

Итого по **поверхности PR** (MVC guard на `LogOn` / `ExternalLogOn`): основные негативы и happy path **на HTTP** закрыты прогоном pytest на согласованном стенде.

## 2. Чего в этом автонаборе нет и что это значит

### 2.1 Политики (AC A4, TC-FO-GUARD-06)

**Что имелось в виду:** после входа действия, завязанные на PolicyRules / разрешения, не регрессировали.

**Почему не в pytest-файле:** это уже **продуктовый smoke** (роли, конкретные экраны, набор прав). В спеке это вынесено в отдельный TC-06, не в контрактные тесты guard’а логона.

**Как закрывать:** чеклист QA / короткий ручной прогон после деплоя, либо отдельная автоматизация не в `frontoffice_login_guard`.

---

### 2.2 SQL / БД (AC A3, формулировка про `GetUserByLoginOrEmail`)

**Что имелось в виду:** для «несуществующего» логина при готовом Bloom нет лишних обращений к БД за пользователем (в тикете упоминалась процедура уровня политик/данных).

**Почему не в pytest:** из клиента **нельзя** надёжно увидеть число вызовов хранимок. Нужны **SQL trace**, профайлер или серверные логи на стенде с согласования.

**Как закрывать:** ручной прогон с trace по сценарию TC-05 при **готовом** Bloom; сравнение «до/после» по методологии QA.

---

### 2.3 TC-FO-GUARD-07 — Regal LP, «медленное переключение» (тикет; **вне PR 15493**)

**Позиция разработки:** по обсуждению (Gleb Grafa) **PR 15493 здесь ничего не «чинил»** — формулировка в переписке: *«это не решал»*. Guard на MVC `LogOn` / `ExternalLogOn` **не претендует** на устранение симптома медленного переключения аккаунта в Regal, если узкое место не этот путь.

**Что тогда такое TC-07 в тест-плане:** отдельная линия расследования тикета — повторить **точное** действие в Regal LP, параллельно SQL/trace, зафиксировать маршрут и вынести в follow-up **[УТОЧНИТЬ]**. Это **не** критерий приёмки PR 15493.

**Что делает pytest:** при необходимости — дым GET LogOn и опциональный replay через `FO_GUARD07_*` (если заранее снят запрос из Network). Полноценная автоматизация Regal без этого — **не** в `frontoffice_login_guard`.

**Как закрывать по смыслу TC-07:** вручную в продукте + trace; отдельный тикет/задача, если симптом остаётся после выката guard’а логона.

---

### 2.4 Рестарт пула / приложения (TC-08, AC A6 частично)

**Что имелось в виду:** после **рестарта** IIS/пула сразу логин; при сбое инициализации Bloom — логи, процесс не падает.

**Что делает pytest:** только «холодный» GET/POST **без** реального рестарта сервера (инфраструктурная операция).

**Как закрывать:** ручной или pipeline-шаг с рестартом на тестовом стенде + просмотр логов при искусственном сбое `GetUsers()` (если такой сценарий доступен).

---

### 2.5 Строгий Bloom → 401 без флага

**Что имелось в виду:** при **готовом** Bloom ответ на заведомо чужой логин — **401** от precheck, а не «мягкий» 200 с формой.

**Почему флаг `FO_STRICT_BLOOM_401`:** до прогрева Bloom фильтр в **безопасном режиме** пропускает запросы дальше; один и тот же тест без флага допускает **401 или 200/302**, чтобы не ломать прогон на «холодном» приложении.

**Как закрывать строго:** прогон с `FO_STRICT_BLOOM_401=1` **после** прогрева Bloom (или отдельный job после ожидания/рестарта).

---

### 2.6 Ветка rate limit «>30 POST с одного IP с разными логинами» (TC-03)

**Что в спеке:** превысить лимит либо **per IP (30/60)**, либо **per IP+login (5/60)**.

**Что делает автотест:** стабильно проверяется вторая ветка (один логин, неверный пароль). Первая ветка (много **разных** логинов, каждый должен **проходить** Bloom precheck, иначе 401 до rate limit) тяжела и избыточна, если вторая ветка уже доказывает наличие `LoginRateLimitAttribute`.

**При необходимости:** отдельный ручной или нагрузочный сценарий по спеке.

## 4. Что ещё осталось пройти (простыми словами и по шагам)

Ниже — **не** дублирование pytest. Это то, что **автотесты сами по себе не доказывают**. Если нужно «закрыть тикет целиком» или «убедиться до конца», вот отдельный чеклист.

---

### Кейс «Политики и права после входа» (TC-FO-GUARD-06, AC A4)

**Зачем:** автотест проверяет логон и guard. Он **не** ходит по экранам и **не** проверяет, что политики (PolicyRules, капча на логине при определённых правилах и т.п.) не сломались.

**Что сделать руками:**

1. Зайти на стенд под **обычным** пользователем (не только «техническая» учётка для API).
2. Выполнить **2–3 привычных действия** после входа: то, что у вас в чеклисте для FO (например открыть нужный раздел, сменить контекст там, где раньше ловили баги).
3. Убедиться, что **нет новых блокировок**, странных редиректов и ошибок прав, которых не было до выката.

**Когда считать ок:** поведение совпадает с эталоном «как до PR» для тех же ролей (или зафиксировано осознанное отличие).

---

### Кейс «База данных: нет лишних запросов за несуществующим пользователем» (AC A3, кусок TC-FO-GUARD-05)

**Зачем:** в тикете звучала идея: если логин **точно не существует**, не надо гонять лишние запросы к БД. **Браузер этого не видит** — видит только HTTP-код.

**Что сделать:**

1. Договориться с кем-то, у кого есть доступ к **SQL trace / профайлеру** на тестовом стенде (или включить согласованный trace на короткое время).
2. Дождаться, пока приложение **прогреет Bloom** (после старта подождать минуту-две или по регламенту стенда).
3. Один раз отправить логон с **заведомо выдуманным** логином (таким, которого точно нет в системе).
4. На trace посмотреть: **не** полезли ли лишний раз в хранимки за пользователем так, как «до оптимизации».

**Когда считать ок:** по методологии QA / разработки зафиксировано «ожидаемо / лучше чем было»; если trace недоступен — этот пункт **остаётся открытым** или закрывается экспертным мнением.

---

### Кейс «Медленное переключение в Regal LP» (TC-FO-GUARD-07)

**Зачем:** это **про другой симптом**, не про «сработал ли rate limit на LogOn». Разработка прямо говорила, что **этот PR тут не целился** («это не решал»).

**Что сделать, если тикет всё ещё про боль Regal:**

1. Сесть в **реальный** сценарий Regal LP: сделать **ровно то действие**, на котором пользователи жалуются (не абстрактный «ещё раз LogOn», а то переключение, которое в тикете).
2. Параллельно (если можно) — **SQL/trace** или HAR в браузере.
3. Записать вывод: узкое место **в MVC LogOn** или **в другом месте** (API, фоновые запросы, смена контекста без этого URL).

**Когда считать ок для самого PR 15493:** ничего — это **не** критерий приёмки PR.  
**Когда считать ок для тикета:** либо симптом ушёл по другой задаче, либо заведён **отдельный** follow-up с понятным корнем.

---

### Кейс «Рестарт приложения и Bloom» (TC-FO-GUARD-08, AC A6)

**Зачем:** автотест имитирует «холодный» запрос без настоящего рестарта IIS/пула. В спеке — именно **рестарт**, потом сразу логин.

**Что сделать:**

1. На **тестовом** стенде (не прод без разрешения) выполнить **рестарт** пула приложения или то, что у вас принято (по согласованию с админами).
2. Сразу после старта попробовать логон **легитимным** пользователем и **один раз** с выдуманным логином.
3. Убедиться: **нет 500**, нет массовых ложных «пользователя не существует» из-за неготового Bloom (по ожиданиям спеки — в безопасном режиме Bloom «не режет» до готовности).
4. Если есть доступ к логам при ошибке загрузки пользователей для Bloom — глянуть, что приложение **не падает** (AC A6).

---

### Кейс «Строго проверить 401 от Bloom» (дополнение к TC-05)

**Зачем:** в обычном прогоне pytest допускает разные коды, чтобы не падать на «холодном» сервере. **Строгая** проверка — отдельный шаг.

**Что сделать:**

1. Убедиться, что приложение **поработало** после старта (Bloom успел подгрузиться).
2. Запустить тот же pytest с переменной окружения **`FO_STRICT_BLOOM_401=1`** (см. шапку `test_frontoffice_login_guard.py`).
3. Если тест падает — разобрать: Bloom ещё не готов, или логин случайно «попал» в Bloom (редко), или другая причина.

---

### Кейс «Лимит 30 запросов с одного IP на разных логинах» (вторая ветка TC-FO-GUARD-03)

**Зачем:** в спеке написано «или больше 30 POST с одного IP». Автотест проверяет **другую** ветку («больше 5 с одним логином»), её достаточно, чтобы убедиться, что лимит вообще есть.

**Что сделать, если принципиально нужна именно ветка 30/IP:**

1. Вручную или отдельным скриптом отправить **больше 30** POST на `LogOn` **с разных логинов**, но так, чтобы каждый логин **не отсекался** Bloom’ом на 401 **до** счётчика rate limit (на практике это неудобно: случайные логины отсекаются).
2. Проще **согласовать с командой**, что для приёмки достаточно проверенной ветки «5 запросов с одним логином → 429».

## 5. Минимум vs «всё по тикету»

| Вопрос | Минимум для «PR 15493 сделан по guard логона» | Плюс к этому, если нужен «полный тикет» |
|--------|-----------------------------------------------|----------------------------------------|
| Автотесты `qa/frontoffice_login_guard` | Прогнать и зелёные на целевом стенде | — |
| Политики / экраны после входа | По желанию узкий smoke | Полный TC-06 по регламенту |
| SQL / лишние вызовы | Не обязательно для merge PR | Нужно для доказательства A3 «в цифрах» |
| Regal / медленное переключение | **Не** часть приёмки PR | Отдельное расследование TC-07 |

Если нужно одно предложение: **автотесты закрывают «слой логона и лимитов». Всё про политики, базу, Regal и рестарт — отдельные осознанные прогоны по таблице выше.**
