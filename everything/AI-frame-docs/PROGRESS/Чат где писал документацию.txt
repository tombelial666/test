Как это лучше строить

Нужен не фреймворк “по репозиториям”, а фреймворк из 4 слоёв:

1. People context layer

Для каждого члена команды хранится state/context document, не “биография”, а рабочий профиль.

Примерно так:

person:
  id: sergey
  role: backend_dev
  domains:
    - etna-trader
    - oms
    - account-opening
  usually_works_with:
    repos:
      - etna-trader
      - oms-services
      - qa-automation
    services:
      - oms
      - ams
      - notification-service
    artifacts:
      - swagger
      - sql
      - logs
      - configs
  typical_tasks:
    - feature implementation
    - bugfix
    - root cause analysis
    - PR review
  useful_context:
    - architecture docs
    - service map
    - frequent folders
    - important tickets

Это нужно не для красоты, а чтобы orchestrator понимал:

чей контекст подтянуть
какие источники открыть первыми
какие документы и части системы этому человеку обычно релевантны
2. Work item layer

Главная сущность должна быть не repo, а task/work item.

То есть задача описывается так:

work_item:
  id: EXT-12345
  title: Fix margin calculation inconsistency
  owners:
    - artem
    - sergey
  related_repositories:
    - etna-trader
    - oms-services
    - qa-automation
  related_services:
    - oms
    - risk-manager
  related_artifacts:
    - swagger
    - logs
    - sql
    - test-cases
  primary_domain: trading

Тогда система строится вокруг ответа на вопрос:

“Что нужно поднять для этой задачи?”, а не
“В каком репозитории мы сейчас стоим?”

Это намного правильнее для вашей реальности.

3. Repository index layer

Нужна глобальная индексация рабочих репозиториев, но не плоская.

Не просто:

repo A
repo B
repo C

А с разбиением на:

репозиторий
домен
сервис
директорию
тип артефакта
владельцев
связи

Пример:

repository:
  name: etna-trader
  domains:
    - trading
    - account-opening
    - clearing
  contains:
    services:
      - oms
      - ams
      - notification-service
    artifact_types:
      - code
      - configs
      - docs
      - tests
  hotspots:
    - src/Etna.Trading.Oms
    - src/AccountOpening
    - src/Clearing

Для огромных реп не индексируется только верхний уровень.
Нужна иерархическая индексация:

repo
subsystem
service
module/folder
artifact type

Потому что “Trader” сам по себе слишком большой и бесполезный как единица контекста.

4. Orchestrator layer

Оркестратор тогда работает не как “выбери один репозиторий”, а как:

принимает задачу
определяет домен
определяет участников
поднимает их context docs
определяет затронутые сервисы
находит релевантные репозитории и подпапки
собирает единый task context
только потом запускает pipeline: QA / Dev / Docs / RCA

То есть routing должен быть не repo-based, а:

routing:
  by_work_item: true
  by_person_context: true
  by_service_map: true
  by_repository: secondary

И это, по сути, главный архитектурный сдвиг.

Как думать правильно в вашем случае

У тебя есть 3 возможные модели.

Плохая модель

Один pipeline = один repo

Не подойдёт, потому что:

задачи живут в нескольких репах
контекст разрывается
тесты и код не всегда вместе
документация может быть отдельно
Средняя модель

Один pipeline = один domain

Лучше, но тоже не идеально, потому что внутри домена может быть много репозиториев и разных людей.

Нормальная модель

Один pipeline = один work item

Это то, что вам ближе всего.

Пайплайн строится вокруг:

задачи
человека
домена
сервисов
артефактов

А репозиторий — это только один из источников данных.

2. Active state

Меняется часто:

какие документы сейчас важны

Тогда фреймворк сможет подтягивать не только “кто ты вообще”, но и “что у тебя сейчас в фокусе”.

Как решать мульти-репо задачу технически

Для одной задачи у тебя должен собираться task bundle.

Например:

task_bundle:

    - artem
    - sergey

        - src/OMS
        - src/Clearing
    - name: qa-automation
      relevant_paths:
        - tests/account-opening
  artifacts:
    - swagger
    - logs
    - sql
    - configs
    - docs

То есть система не просто знает “какие репы есть”, а умеет на задачу собрать минимальный релевантный набор.

Это критично, иначе LLM будет тонуть в лишнем контексте.

конкретные папки
конкретные конфиги
конкретные тесты
конкретные Swagger/logs/SQL

И уже из этого строится Task Bundle. Это и есть нормальный способ работы с легаси в мульти-репо и huge-repo среде. В PBI это отдельно описано как иерархическая индексируемая модель для больших репозиториев.

Зачем это усложнение вообще нужно

Потому что без иерархии возникают 4 проблемы:

Слишком много шума
Если взять весь legacy repo целиком, AI и человек тонут в нерелевантном коде.
Неправильный routing
Задача может быть в одном репо, но реально затрагивать 2–3 подсистемы и часть автотестов в другом месте.
Потеря контекста между ролями
Dev знает, где код, QA знает, где тесты, а Lead — только тикет. И без индекса каждый заново собирает карту мира.
Сложно работать с “историческим” кодом
В легаси важен не весь репозиторий, а hotspots: старые модули, нестабильные директории, важные конфиги, связанные артефакты.
Как я предлагал с этим работать
1. Ввести Repository Index

Для каждого большого репозитория хранить YAML-описание:

роль репозитория
размер/тип
подсистемы
сервисы
пути к коду
пути к тестам
пути к докам
владельцев/команды
ограничения доступа

Примерно так и было предложено в PBI и в архитектуре.

2. Ввести Service Map

Отдельно держать связь:

сервис → какие репы
сервис → какие конфиги
сервис → какие API
сервис → какие логи
сервис → какие SQL/тесты

Это нужно, потому что в легаси логика часто размазана не по одной папке, а по нескольким слоям. Архитектура это тоже выделяет отдельно.

3. Строить Task Bundle узко

Task Carrier может быть широким и хранить всё, что известно по задаче, а Task Bundle должен быть узким: только нужные папки, сервисы и артефакты под конкретный запуск. Это прям один из центральных принципов.

4. Для легаси добавить hotspots

То есть индекс должен знать не только структуру, но и “опасные зоны”:

старые нестабильные модули
чувствительные конфиги
известные проблемные директории
места, где часто бывают регрессии

Это уже следующий шаг после базовой иерархии, но он очень полезен для QA и RCA.

Как это будет выглядеть на практике

Допустим, задача “ошибка в маржинальном расчёте OMS”.

Без иерархии система видит:

репо etna-trader

С иерархией система видит:

repo: etna-trader
subsystem: OMS
service: risk-manager / margin logic
code paths:
src/Etna.Trading.Oms/...
related tests:
qa-automation/tests/oms/...
artifacts:
swagger
logs
sql query
owner/team:
backend dev + QA

И уже на основе этого собирается bundle, а не тащится весь трейдер целиком.

Как это рассказать на дейлике

Можно так:

Для работы с легаси я предложил не индексировать большие репозитории плоско, а разбивать их иерархически: репозиторий → подсистема → сервис → конкретные пути и артефакты. Это нужно, чтобы под задачу собирать только релевантный контекст, а не тащить весь legacy-монолит. За счёт этого orchestration будет точнее, а Dev и QA смогут быстрее поднимать нужную часть системы.
